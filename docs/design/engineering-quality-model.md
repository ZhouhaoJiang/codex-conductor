# Conductor 工程质量模型

状态：已接受
日期：2026-07-14

## 背景

Conductor Lite 的目标是让强模型直接解决问题，而不是用细密的流程拦截每一步。
但“轻量”不能被理解为不要设计、计划、规范、测试或验收。真正要去掉的是无差别
治理：不论任务大小都生成同样的文档、重复状态，或把多个 review 视角变成多个
强制关卡。

## 决策

把工程质量拆成五个彼此相关、证据不同的部分：

| 部分 | 发生时机 | 主要产物或证据 |
| --- | --- | --- |
| 设计 | 存在需要长期保留的边界或取舍时 | 设计文档、RFC 或仓库原生产物 |
| 计划 | 存在多阶段、协作、风险或长期状态时 | 可更新的短计划文档 |
| 编码 | 实现开始前先确定适用规范，并在编写中执行 | 与项目一致的代码和静态检查结果 |
| 测试 | 存在可执行单元行为时 | 仓库测试目录中的真实测试文件及运行结果 |
| 功能验收 | 实现后按需求场景检查真实路径 | UI、API、CLI 或集成路径的实际行为 |

Review 不再作为第六条流水线，而是对以上结果切换视角进行复核：设计/计划视角、
代码视角、测试视角和功能视角可以由同一个 reviewer 在一轮聚焦检查中完成；只有
独立性确实能提高可信度时，才拆成有边界的原生 subagent；只有用户明确要求持久、
可见的 reviewer 时，才创建额外 session/thread。

## 文档尺度

- 架构、公共 API、数据模型、安全边界、跨组件行为和重要取舍需要设计文档。
- 多阶段、多人协作、高风险或需要跨会话维护状态的工作需要计划文档。
- 小而局部、可逆的修改可以只在会话里保留短计划，路径清楚时也可以直接做。
- 优先使用仓库已有的 RFC、issue、design、plan 模板和目录；不为同一状态再建一套
  ledger。
- 文档是否重要取决于它保存的决策和协调价值，而不是篇幅。

## 编码规范的选择顺序

1. 用户明确要求和仓库规则优先，包括 `AGENTS.md`、项目文档、formatter、linter、
   编译配置和邻近代码风格。
2. 仓库没有规定时，采用语言或框架的官方惯例。
3. 仍有空白时，选择适合该语言、成熟且大规模验证过的规范；Google Style Guide
   是合适时的强默认候选，但不能覆盖已有项目约定。

规范应在写代码前选定，并在实现过程中使用。Code review 的职责是发现残余风险，
不是等代码写完后才第一次引入基本规范。也不能为了套用回退规范，擅自安装新的
formatter、修改全局工具链或格式化无关文件。

## 测试与验收

- “有单元测试”意味着测试存在于仓库真实测试文件中；临时脚本、单条 shell 命令、
  手工请求和复现步骤属于诊断或验证证据，不叫单元测试。
- 优先扩展最近的现有测试文件；新的职责边界需要独立归属时再创建新文件。
- 文档、元数据等没有可执行单元边界的修改不强造假单测，而应执行与产物匹配的验证。
- 功能验收从用户需求推导场景，并走真实 UI、API、CLI、集成或运行时路径。
- 单测、类型检查和构建通过都不能替代功能验收；真实路径无法运行时必须明确说明。

## Review 视角

- 设计/计划：需求、边界、取舍、范围和执行状态是否一致。
- 代码：正确性、安全性、可维护性以及是否符合选定规范。
- 测试：真实测试文件是否覆盖关键行为和可信失败，而非贴着实现细节写。
- 功能：用户可见路径及关键状态是否按需求工作。

这些是思考视角，不是默认的多智能体编排，也不是四轮强制门禁。一轮检查可以覆盖
所有相关视角；任务不相关的视角可以不做。

## 在插件中的落点

- `conductor-lite/SKILL.md` 只保留按需读取入口，避免每次激活都注入长规则。
- `conductor-lite/references/engineering-quality.md` 保存可复用的完整默认判断。
- `debugging`、`refactor` 和 `frontend` 只保留各自场景必须知道的测试或验收区别。
- 不新增 always-on hook，不自动创建文档，不默认发起 review subagent 或 session。

## 结果

这个模型把设计、计划和质量证据保留下来，同时允许 Codex 根据任务规模选择最小充分
形式。它是一组默认判断，不宣称对所有项目和所有任务绝对正确。
